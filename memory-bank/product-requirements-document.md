# AI辅助求职平台 - 产品需求文档（PRD）

## 1. 文档概述

| 字段 | 内容 |
|------|------|
| 产品名称 | AI求职助手（暂定） |
| 版本 | V1.0 |
| 目标用户 | 互联网行业实习/校招求职者（在校生、应届毕业生） |
| 产品形态 | Web应用（响应式，兼容移动端浏览器） |
| 核心价值 | 一站式管理求职全流程：岗位匹配→简历优化→投递管理→面试准备→面试复盘 |

---

## 2. 用户画像

| 画像 | 描述 |
|------|------|
| 典型用户A | 大三学生，计算机专业，寻找暑期实习，有1-2段项目经历，不确定简历如何针对性修改 |
| 典型用户B | 应届硕士，转行互联网产品岗，需要从大量岗位中筛选匹配项，面试经验不足 |
| 典型用户C | 大四学生，秋招海投阶段，投递数量多，难以管理进度，需要针对不同岗位定制简历 |

## 4. 详细功能需求

### 4.1 模块一：简历管理

#### 4.1.1 简历上传

| 项目 | 说明 |
|------|------|
| 支持格式 | PDF（优先）、Word（.docx）、图片（OCR识别） |
| 文件大小 | 单文件不超过10MB |
| 解析内容 | 个人信息、教育背景、实习/项目经历、技能清单、获奖情况 |
| 解析结果展示 | 结构化展示解析内容，用户可手动修正 |

#### 4.1.2 简历结构化存储

解析后的简历数据结构：

```json
{
  "basic_info": {
    "name": "",
    "phone": "",
    "email": "",
    "education": [
      {
        "school": "",
        "degree": "",
        "major": "",
        "gpa": "",
        "start_date": "",
        "end_date": ""
      }
    ]
  },
  "experiences": [
    {
      "id": "",
      "type": "internship | project | research | competition",
      "company_or_org": "",
      "role": "",
      "duration": "",
      "descriptions": [""],
      "skills_involved": [""],
      "metrics": [""]
    }
  ],
  "skills": [""],
  "awards": [""],
  "certifications": [""]
}
```

#### 4.1.3 多版本管理

- 用户可保存多份简历版本（如"产品岗版本"、"运营岗版本"）
- 支持基于某版本创建新版本
- 版本间可对比差异

### 4.2 模块二：求职意向与岗位匹配

#### 4.2.1 求职意向设置

| 筛选维度 | 选项 |
|----------|------|
| 岗位方向 | 前端开发、后端开发、算法工程师、数据分析、产品经理、运营、测试、设计等（支持多选） |
| 求职类型 | 暑期实习、日常实习、秋招、春招、社招 |
| 意向城市 | 北京、上海、深圳、杭州、广州、成都、南京等（支持多选） |
| 公司规模 | 大厂（BAT/TMD等）、中厂（独角兽/上市公司）、初创公司、不限 |
| 行业偏好 | 电商、社交、金融科技、教育、游戏、企业服务、AI等 |
| 薪资期望 | 区间选择（实习按日薪，全职按月薪） |

#### 4.2.2 岗位信息抓取

数据源覆盖：

| 来源类型 | 具体平台 |
|----------|----------|
| 公司官网 | 字节跳动、腾讯、阿里巴巴、美团、京东、百度、网易、快手、小红书、拼多多等Top50互联网公司招聘官网 |
| 招聘平台 | 牛客网、实习僧、Boss直聘（公开信息）、官方公众号 |
| 更新频率 | 每日定时抓取（凌晨2:00），用户手动刷新时实时抓取 |

抓取的岗位数据结构：

```json
{
  "job_id": "",
  "company": "",
  "position_name": "",
  "department": "",
  "job_type": "intern | campus | social",
  "location": [],
  "jd_content": "",
  "requirements": [""],
  "preferred": [""],
  "salary_range": "",
  "post_date": "",
  "deadline": "",
  "source_url": "",
  "company_size": "",
  "industry": ""
}
```

#### 4.2.3 智能匹配算法

匹配维度与权重：

| 维度 | 权重 | 匹配逻辑 |
|------|------|----------|
| 技能匹配 | 30% | 用户技能与JD要求技能的重合度 |
| 经历相关性 | 25% | 用户经历与岗位职责的语义相似度 |
| 硬性条件 | 20% | 学历、城市、求职类型是否满足 |
| 行业/方向 | 15% | 用户意向与岗位方向的吻合度 |
| 优先条件 | 10% | JD中"加分项"与用户背景的匹配 |

匹配结果展示：
- 匹配度百分比（如92%）
- 匹配/不匹配项的具体说明
- 按匹配度降序排列，支持筛选和排序
- 每个岗位卡片展示：公司logo、岗位名、部门、城市、匹配度、截止日期

### 4.3 模块三：简历优化

#### 4.3.1 针对性简历修改方案

用户点击某个匹配岗位后，系统生成针对该岗位的简历修改方案：

**修改方案结构：**

```
一、整体策略
   - 该岗位核心考察能力：[列出3-5项]
   - 你的简历当前突出能力：[列出当前突出项]
   - 调整方向：[概述需要的调整]

二、逐条修改建议
   经历1：[经历标题]
   ├── 建议：保留/删除/重写/调整顺序
   ├── 原文："负责用户增长相关数据分析工作"
   ├── 改为："基于SQL+Python搭建用户增长归因模型，覆盖5条核心转化路径，
   │         驱动产品迭代后次日留存率提升12%"
   ├── 修改理由：原文缺少技术栈关键词和量化结果，
   │            该岗位JD强调数据驱动和业务指标意识
   └── 关键词命中：SQL、Python、归因模型、留存率

   经历2：...

三、关键词补充建议
   - JD中出现但简历缺失的关键词：[列表]
   - 建议添加位置和方式

四、经历取舍建议
   - 建议突出的经历：[哪些经历与岗位强相关]
   - 建议弱化/删除的经历：[哪些经历与岗位无关或拉低匹配度]
   - 如有空间，建议补充的内容方向

五、排版顺序建议
   - 建议的经历排列顺序（按相关性）
```

#### 4.3.2 一键生成优化简历

- 基于修改方案自动生成新版本简历
- 用户可逐条接受/拒绝修改建议
- 修改前后对比视图（diff模式）
- 生成后可导出为PDF

#### 4.3.3 修改质量保障

- 保持用户原有经历的真实性，不捏造内容
- 优化表述方式、突出关键词、量化结果
- 如用户经历确实缺乏相关性，提示"建议补充XX类型经历"而非凭空编造

### 4.4 模块四：投递管理

#### 4.4.1 投递表格

用户投递简历后（手动标记或系统检测），自动生成投递记录：

| 字段 | 说明 | 类型 |
|------|------|------|
| 公司 | 公司名称 | 文本 |
| 岗位 | 岗位名称 | 文本 |
| 部门 | 所属部门/事业群 | 文本 |
| 投递时间 | 投递日期 | 日期 |
| 当前进度 | 已投递/简历筛选中/笔试/一面/二面/三面/HR面/Offer/已拒 | 枚举 |
| JD链接 | 原始招聘链接 | URL |
| Base地 | 工作城市 | 文本 |
| 优先级 | P0(必冲)/P1(重点)/P2(保底)/P3(海投) | 枚举 |
| 使用简历版本 | 投递时使用的简历版本 | 关联 |
| 备注 | 用户自由备注 | 文本 |
| 薪资信息 | 已知薪资范围 | 文本 |
| 截止日期 | 岗位截止投递日期 | 日期 |
| 面试时间 | 各轮面试时间记录 | 日期列表 |

#### 4.4.2 表格功能

- 支持按公司、进度、优先级、时间等维度筛选和排序
- 支持批量操作（批量更新进度、批量设置优先级）
- 数据看板：投递总数、各阶段转化率漏斗图、本周新增投递数
- 日历视图：展示面试时间安排
- 导出功能：支持导出为Excel/CSV
- 提醒功能：面试前24小时提醒、岗位截止前3天提醒

#### 4.4.3 进度自动更新

- 用户完成模拟面试或上传面试录音后，提示更新对应岗位进度
- 用户收到面试邀请邮件时（未来版本，需授权邮箱），自动更新进度

### 4.5 模块五：模拟面试

#### 4.5.1 面试结构

每个岗位提供三轮模拟面试：

| 轮次 | 名称 | 考察重点 | 时长 | 题量 |
|------|------|----------|------|------|
| 第一轮 | 技术/业务基础面 | 做了什么、怎么做的 | 30-45分钟 | 8-12题 |
| 第二轮 | 深度思维面 | 为什么这么做、如何权衡 | 30-45分钟 | 6-10题 |
| 第三轮 | HR综合面 | 稳定性、价值观、职业规划 | 20-30分钟 | 6-8题 |

#### 4.5.2 各轮面试详细设计

**第一轮：技术/业务基础面**

题目来源：
- 简历深挖（60%）：针对用户简历中的具体经历提问
  - "这个项目中你具体负责哪个模块？"
  - "遇到XX问题时你是怎么解决的？"
  - "这个指标提升12%是怎么计算的？"
- 八股/基础知识（40%）：根据岗位方向出题
  - 技术岗：数据结构、算法、操作系统、计算机网络、数据库等
  - 产品岗：产品设计方法论、数据分析、用户研究等
  - 运营岗：活动策划、数据指标、用户增长等

**第二轮：深度思维面**

题目来源：
- 简历深挖（50%）：追问决策逻辑
  - "为什么选择这个技术方案而不是XX？"
  - "如果重新做这个项目，你会怎么改进？"
  - "这个方案的局限性是什么？"
- 开放性问题（30%）：考察思维深度
  - 系统设计题（技术岗）
  - 产品设计题（产品岗）
  - 案例分析题（运营岗）
- 行业认知（20%）：
  - "你怎么看待XX行业的发展趋势？"
  - "竞品分析"

**第三轮：HR综合面**

题目类型：
- 稳定性考察：为什么选择我们公司？职业规划？期望薪资？
- 价值观考察：团队合作中的冲突如何处理？最大的挫折？
- 综合素养：自我介绍、优缺点、反问环节
- 压力测试：如果拿到多个offer怎么选？为什么不读研/为什么转行？

#### 4.5.3 面试官风格

| 风格 | 特点 | 适用场景 |
|------|------|----------|
| 温和引导型 | 会给提示，追问温和，适合初次练习 | 面试新手 |
| 注重细节型 | 对数据、时间线、技术细节追问很深 | 准备大厂面试 |
| 严格压力型 | 频繁challenge，语气直接，会打断 | 压力面试训练 |
| 发散思维型 | 喜欢追问"还有呢"、"换个角度呢" | 产品/策略岗 |
| 沉默寡言型 | 反馈少，表情平淡，考验自驱表达 | 模拟真实场景 |

#### 4.5.4 面试交互方式

- 文字对话模式（V1.0）：用户打字回答，AI文字追问
- 语音对话模式（V1.5）：用户语音回答，AI语音提问（更接近真实面试）
- 每轮面试结束后生成评估报告：
  - 各题得分（1-10分）
  - 回答亮点
  - 改进建议
  - 参考答案/答题框架
  - 综合评级：通过/待提升/不通过

#### 4.5.5 面试复盘（真实面试）

**录音上传与处理：**

| 步骤 | 说明 |
|------|------|
| 上传 | 支持mp3、m4a、wav格式，单文件不超过500MB |
| 转写 | 语音转文本（ASR），区分面试官和候选人 |
| 结构化 | 按问答对整理，标注问题类型 |
| 分析 | 生成复盘报告 |

**复盘报告内容：**

```
一、面试基本信息
   - 公司/岗位/面试轮次/时长

二、问答还原
   Q1: [面试官问题]
   A1: [你的回答]
   评价: [回答质量评估]
   改进: [更好的回答方式]

三、整体表现分析
   - 表达流畅度
   - 逻辑清晰度
   - 技术深度/业务理解
   - 亮点时刻
   - 失误/卡壳点

四、后续准备建议
   - 需要补充学习的知识点
   - 需要重新梳理的项目经历
   - 下一轮面试预测重点

五、投递表格更新建议
   - 建议将进度更新为：[XX]
   - 预判结果：[通过概率评估]
```

## 5. 非功能需求

### 5.1 性能要求

| 指标 | 要求 |
|------|------|
| 简历解析 | 上传后30秒内完成解析 |
| 岗位匹配 | 意向设置后60秒内返回匹配结果 |
| 简历优化方案 | 点击岗位后45秒内生成方案 |
| 模拟面试响应 | 用户回答后5秒内给出下一个问题 |
| 录音转写 | 1小时录音在10分钟内完成转写和分析 |
| 页面加载 | 首屏加载时间 < 3秒 |

### 5.2 安全与隐私

- 用户简历数据加密存储（AES-256）
- 面试录音处理完成后可选择删除原始音频
- 不将用户简历数据用于模型训练（明确告知用户）
- 支持用户一键导出/删除所有个人数据（GDPR合规）
- 登录支持手机号+验证码、微信扫码

### 5.3 可用性

- 响应式设计，适配桌面端和移动端浏览器
- 核心操作路径不超过3步
- 新用户引导流程（onboarding）
- 关键操作有加载状态和进度提示

```
├──────────┬──────────┬──────────┬────────────────┤

│   简历解析 | 匹配算法 | 简历优化 | 面试对话 | ASR  │
├─────────────────────────────────────────────────┤
│              数据层                               │
│   PostgreSQL | Redis | OSS(文件) | ES(搜索)      │
├─────────────────────────────────────────────────┤
│              爬虫服务                             │
│   定时任务 | 反爬处理 | 数据清洗 | 增量更新        │
└─────────────────────────────────────────────────┘
```

### 6.1 关键技术选型

| 模块 | 技术方案 |
|------|----------|
| 简历解析 | LLM结构化提取 + PDF解析库（pdf.js） |
| 岗位抓取 | Scrapy/Playwright + 代理池 + 定时调度 |
| 智能匹配 | 向量相似度（Embedding）+ 规则引擎混合 |
| 简历优化 | LLM（GPT-4级别）+ Prompt Engineering |
| 模拟面试 | LLM多轮对话 + 角色设定Prompt |
| 语音转写 | Whisper / 第三方ASR服务 |
| 语音面试 | TTS + ASR + LLM实时对话 |

## 10. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 岗位抓取被反爬 | 数据源不可用 | 多源备份、代理池、降低频率、接入官方API（如有） |
| LLM生成质量不稳定 | 简历优化/面试质量差 | Prompt迭代优化、人工审核样本、用户反馈闭环 |
| 简历隐私顾虑 | 用户不愿上传 | 明确隐私政策、本地解析选项（V2.0）、数据加密 |
| 岗位信息时效性 | 推荐已关闭岗位 | 每日更新、用户反馈标记、过期自动下架 |
| 模拟面试与真实差距大 | 用户觉得没用 | 持续收集真实面经优化题库、支持录音复盘对比 |
